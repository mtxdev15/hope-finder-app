01

Launch-gating decisions

Four product decisions gate a real launch. Three are resolved; one needs the founder + legal. Everything downstream in this doc assumes these answers.

1

Content source

Resolved

Server-generated via the Claude API, with verses from the American Bible API. The prototype's in-browser window.claude.complete does not exist in production — generation moves to a Cloudflare Worker. The authored fallback bank stays as the safety net (see §6).

2

Accounts & persistence

Resolved

Real auth + real persistence on Supabase (Postgres + Auth + RLS). Every localStorage key in the prototype maps to a table or a user-preferences column (see §4). Guest/local mode persists until first sign-in, then migrates.

3

Reminders & notifications

Resolved

Server-scheduled. A Cloudflare Cron Worker drives the daily reminder + journey check-ins, honoring quiet hours and per-channel toggles. Web Push for the PWA now; APNs for the native app later (see §8).

4

Scripture licensing

Decision needed

The app ships KJV / WEB / ASV (public domain) today. NKJV, NIV, NLT — marketed on the homepage and used across the SEO pages — are copyrighted and require licenses. Decision: license them, or swap to public-domain text and correct the "six translations" claim (see §7).

02

Target architecture

A static-first Astro site on Cloudflare Pages, a thin Worker API for everything dynamic, Supabase as the system of record, and two external services behind the Worker so no key ever reaches the browser.

Client Astro site + PWA
Static marketing/SEO pages (pre-rendered) + the app shell (Today, Journey, Vault, The Word, Account, Find a Church). Service worker for offline + Web Push. This is the current HTML, ported to Astro components.
↓   fetch / JSON   ↓
Edge API Cloudflare Workers
One Worker (or a small set) exposing /api/*: daily word generation, Bible passage proxy + cache, vault sync, journey state, reminders registration, ratings, waitlist. Holds all secrets. Cron Triggers fire scheduled notifications.
↓   Postgres · REST · RPC   ↓
Data Supabase
Postgres (system of record), Auth (magic link + OAuth), Row Level Security, Storage (card-studio exports, avatars), Realtime (optional, for cross-device vault sync).
↑   called only from the Worker   ↑
External Claude API & American Bible API
Anthropic Claude generates the verse-reflection-declaration-prayer set and journey days. The American Bible API serves verbatim Scripture by translation. Both are server-side only, rate-limited, and cached in Postgres.
Why this shape

The prototype is already static HTML with client logic — Astro + Cloudflare Pages is a near-direct port with zero server runtime to manage. Workers give you a single secret boundary and cron in the same platform. Supabase covers auth, DB, and storage without standing up infrastructure. Nothing here requires a long-running server.

Schema (abridged DDL)

create table profiles (
  id            uuid primary key references auth.users(id) on delete cascade,
  first_name    text,
  country       text,
  home_church_id text,
  role          text default 'member',   -- 'member' | 'admin'
  prefs         jsonb default '{}',       -- theme, reader display, translation
  last_begin_on date,
  created_at    timestamptz default now()
);

create table vault_items (
  id        uuid primary key default gen_random_uuid(),
  user_id   uuid references profiles(id) on delete cascade,
  type      text,                       -- 'verse' | 'declaration' | 'prayer'
  body      text,
  reference text,                       -- e.g. 'John 1:1 KJV'
  collection text,                      -- e.g. 'Morning Strength'
  is_mine   boolean default false,      -- user-authored vs received
  created_at timestamptz default now()
);

create table journeys (
  id        uuid primary key default gen_random_uuid(),
  user_id   uuid references profiles(id) on delete cascade,
  struggle  text,
  status    text default 'active',     -- 'active' | 'complete'
  day       int default 1,              -- 1..5
  started_at timestamptz default now(),
  unique(user_id, struggle, started_at)
);

create table journey_days (
  journey_id uuid references journeys(id) on delete cascade,
  day        int,                      -- the per-day pacing lock
  unlocked_on date,                     -- "one day per calendar day"
  returned   boolean default false,
  payload    jsonb,                    -- generated verse/reflection/declaration/prayer
  primary key(journey_id, day)
);

create table reminder_settings (
  user_id   uuid primary key references profiles(id) on delete cascade,
  reminder_at text,                    -- '08:00' | custom ISO datetime | 'off'
  tz        text,                       -- IANA, e.g. 'America/New_York'
  push      boolean default true,
  journey   boolean default true,
  updates   boolean default true,
  email     boolean default false,
  quiet_start text, quiet_end text
);

create table ratings (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references profiles(id) on delete set null,
  stars int, chips text[], note text,
  created_at timestamptz default now()
);

create table ios_waitlist (
  email text primary key, user_id uuid, source text,
  created_at timestamptz default now()
);

create table bible_cache (        -- server-side passage cache (replaces db_bible_cache_*)
  key text primary key,            -- '{translation}:{book}:{chapter}'
  translation text, payload jsonb,
  fetched_at timestamptz default now()
);

Row Level Security

Enable RLS on every user table. The pattern is identical across them:

alter table vault_items enable row level security;

create policy "own rows" on vault_items
  for all using (auth.uid() = user_id)
  with check (auth.uid() = user_id);
Guest → member migration

Local mode keeps writing to localStorage until first sign-in. On sign-in, a one-time POST /api/migrate reads the local blobs and inserts them under the new user_id, then clears local. Keep the local keys as the offline cache layer even after migration.

05

Authentication & accounts

Supabase Auth covers the prototype's sign-in screen as-is: passwordless email plus social providers.

  • Email magic link / OTP — matches the existing Signin.html code-entry UI. No passwords to store.
  • OAuth providers — the prototype's provider buttons map to Supabase's Google / Apple providers. Apple Sign-In is required for App Store approval of the native app, so wire it now.
  • Session — Supabase issues a JWT; the browser client stores it. The declare-session flag is replaced by the real session. The "Stay signed in" checkbox maps to session persistence.
  • Profile bootstrap — a Postgres trigger on auth.users insert creates the matching profiles row; first_name is derived from the email local-part exactly as the prototype does today.
  • Account deletion — the Account screen's delete action calls an RPC that cascades (every user table is on delete cascade). Required for App Store compliance.
Admin gating

The check-in notification preview and any internal tooling key off profiles.role = 'admin' — replacing the prototype's localStorage['declare-admin'] flag. Gate admin-only UI server-side, never on a client flag.

06

Content pipeline — Claude + Bible API

The heart of the product: a struggle in, and a verse + reflection + declarations + prayer out. In production this runs server-side, with Scripture kept verbatim and trustworthy.

Generation flow

  1. Client POST /api/daily with { struggle, freeText?, day? }.
  2. Worker builds the prompt (the existing journey-engine.js prompt, ported server-side) and calls the Claude API.
  3. Claude returns structured JSON: a chosen Scripture reference, a reflection, declarations, and a prayer — it never invents verse text.
  4. The Worker resolves that reference's verbatim text from the American Bible API (in the user's translation), checking bible_cache first.
  5. Worker assembles the payload, writes it to journey_days.payload (for journeys) and returns it.
Trust boundary — why this split

Scripture text comes only from the licensed Bible API, never from the model. Claude chooses which verse and writes the pastoral wrapper; the Bible API supplies the words. This keeps Scripture exact and licensable, and is exactly how the prototype's journey-engine.js already reasons.

Worker sketch

// functions/api/daily.ts  (Cloudflare Worker)
export async function onRequestPost({ request, env }) {
  const { struggle, freeText, day } = await request.json();

  // 1 — Claude picks the verse + writes reflection/declarations/prayer
  const claude = await fetch("https://api.anthropic.com/v1/messages", {
    method: "POST",
    headers: {
      "x-api-key": env.ANTHROPIC_API_KEY,
      "anthropic-version": "2023-06-01",
      "content-type": "application/json",
    },
    body: JSON.stringify({
      model: "claude-sonnet-4-5",
      max_tokens: 1024,
      messages: [{ role: "user", content: buildPrompt({ struggle, freeText, day }) }],
    }),
  }).then(r => r.json());

  const draft = parseJSON(claude);          // { reference, reflection, declarations[], prayer }
  if (!draft) return json(fallbackBank(struggle));  // authored safety net

  // 2 — resolve verbatim Scripture (cache-first)
  draft.verse = await getPassage(env, draft.reference, translationFor(request));

  return json(draft);
}

Caching & cost control

  • Bible passages — cache in bible_cache by {translation}:{book}:{chapter}. Passages are immutable; cache effectively forever.
  • Daily word — optionally cache a canonical daily generation per (struggle, date) so repeat opens are free and consistent. Free-text entries always generate fresh.
  • Fallback — keep the authored bank from journey-engine.js. On any Claude timeout, rate-limit, or unparseable response, serve it. The user must never see an empty state.
  • Rate limiting — per-user limit at the Worker (e.g. Cloudflare KV counter) to contain cost and abuse.
07

Scripture licensing

This is decision #4 — the one open item. It is a legal gate, not an engineering task, and it affects both the reader and the marketing pages.

Where it stands today

  • The reader (TheWord.html) ships KJV, WEB, ASV — all public domain — and already gates NKJV / NIV / NLT behind a "streams at launch" toast. Good seam.
  • The homepage markets "six translations… NKJV, NIV, NLT, KJV, WEB, and ASV." That promise currently exceeds the rights held.
  • The 13 SEO pages quote NIV / NLT throughout (hero verses and the new closing verses).
Action required before launch

NKJV — license via HarperCollins Christian (Thomas Nelson). NIV — license via Biblica / Zondervan. NLT — license via Tyndale House. Each has quotation thresholds and API terms. Confirm the American Bible API's plan actually covers the translations you display — many require a per-publisher license even when accessed through an aggregator.

Two clean paths

  1. License & keep the claim. Secure NKJV/NIV/NLT, confirm Bible-API coverage, keep the homepage copy. Budget: per-publisher fees + legal review.
  2. Public-domain launch. Display only KJV/WEB/ASV everywhere (including SEO pages), and change the homepage to "three translations." Zero licensing cost, ships immediately. Add copyrighted translations post-launch behind the same toast seam.

The architecture supports either without change — translation is a parameter to the Bible-API proxy. Only the set you display and the marketing copy move.

08

Reminders & notifications

A scheduled Worker drives the daily reminder and journey check-ins, honoring each user's time, timezone, quiet hours, and per-channel toggles — exactly the controls already built into Account.

How it runs

  • Cron Trigger — a Cloudflare scheduled Worker runs every ~15 min. It queries reminder_settings for users whose reminder_at (in their tz) falls in this window, are outside quiet_start…quiet_end, and have the relevant channel enabled.
  • Delivery now (PWA) — Web Push (VAPID). Store push subscriptions per device. The encouraging check-in copy bank lives in the prototype (CHECKINS in Account) — move it to a checkin_copy table so it's editable without a deploy.
  • Delivery later (native) — APNs for the iOS app. The same scheduler picks the channel by device type; the payload contract stays identical.
  • Day-lock — the journey "one day per calendar day" rule (prototype db_journey_lock) becomes journey_days.unlocked_on, enforced server-side so it can't be bypassed by changing the device clock.
Copy is product, not code

The check-in notifications are written to fight the battle in the mind — naming the lie, speaking the truer thing. Keep them in a table the team can edit, and vary them by journey day. The app icon (brand/icon-512.png, the DB + cross mark) is the notification icon on both platforms.

Deployment sequence

A pragmatic order from empty repo to declareandbelieve.com on the v3.2-redesign branch, then merge.

  1. Supabase project — create it, run the §4 schema, enable RLS, configure Auth (email + Google + Apple), add the profiles bootstrap trigger.
  2. Astro repo — port the prototype into the §3 structure. Marketing/SEO pages pre-render; the app shell hydrates. Wire lib/supabase.ts and lib/api.ts.
  3. Worker API — implement /api/{daily,bible,vault,journey,reminders,rate,migrate} + the Bible cache. Add Worker secrets.
  4. Content pipeline — port the journey-engine.js prompt + fallback bank server-side; resolve the Bible-API account and translation coverage (ties to §7).
  5. Cron Worker — wire the scheduler, Web Push subscriptions, and the editable check-in copy table.
  6. Cloudflare Pages — connect the repo, set env vars, point the build at Astro. Deploy a preview on the v3.2-redesign branch first.
  7. QA on preview — auth, guest→member migration, daily generation + fallback, vault sync, journey day-lock, reminders, ratings, waitlist, Find a Church.
  8. DNS & cutover — map the apex domain, verify, then merge v3.2-redesign → main.
  9. Post-launch — add copyrighted translations (if licensed), then begin the native app (§11).
11

Path to the native iOS app

The web build is the proving ground; SwiftUI is the destination. The data layer is shared, so the native app is a new client over the same Supabase + Worker backend.

  • Same backend — SwiftUI calls the identical /api/* Worker endpoints and Supabase Auth (official Swift SDK). No second backend.
  • Auth parity — Apple Sign-In (already required on web) carries straight over; Supabase issues the same JWT.
  • Notifications — swap Web Push for APNs at the delivery step; the scheduler and copy are unchanged.
  • The waitlist — ios_waitlist is your launch list. Wire the store IDs in rate.js's STORE object (currently TODO) when the app ships.
  • Design fidelity — the prototype's motion system and the DB + cross icon are the visual spec; the responsive web shell already approximates the native layouts.
